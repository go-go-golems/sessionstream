# xgoja Redis/Watermill chatdemo host

This example shows the shape of a template/runtime-package-backed xgoja host where JavaScript owns the chat app, but Go owns distributed infrastructure wiring.

The xgoja spec generates an importable runtime package under `internal/xgojaruntime`. The custom Go host in `cmd/redis-host` creates a Redis Streams Watermill publisher/subscriber pair, injects it into the sessionstream xgoja provider through host services, then attaches the generated xgoja commands.

## Architecture

```text
browser(s)
  <-> generated HTTP/Express command from xgoja
  <-> JS chatbot verb
  <-> sessionstream hub
  <-> Watermill Redis Streams topic
  <-> every server process running hub.run()
  <-> each process fans UI events to its own websocket clients
```

The important semantic choice is Redis fan-out mode: the subscriber is created with an empty `ConsumerGroup`, so every running server process receives each event. A shared consumer group would create work-queue semantics where only one process handles each event.

## Files

- `xgoja.yaml` selects the same JS/assets/providers as the regular chatdemo, adds a second `redis-tools` jsverb source, and emits a `runtime-package` artifact.
- `cmd/redis-host/main.go` wires Redis/Watermill and injects `sessionstream` host options.
- `docker-compose.yml` starts Redis.
- `internal/xgojaruntime` is generated by `make generate`.

## Run

Start Redis:

```bash
make -C examples/goja-redis-chatdemo-server compose-up
```

Generate and run one server:

```bash
make -C examples/goja-redis-chatdemo-server serve HTTP_ADDR=127.0.0.1:18889
```

Open:

```text
http://127.0.0.1:18889/
```

Run two processes against the same Redis topic:

```bash
cd /home/manuel/workspaces/2026-06-12/goja-sessionstream/sessionstream

REDIS_ADDR=127.0.0.1:6379 \
SESSIONSTREAM_BUS_TOPIC=sessionstream.chatdemo.redis \
SESSIONSTREAM_CONSUMER_ID=server-a \
go run ./examples/goja-redis-chatdemo-server/cmd/redis-host \
  serve chatbot serve --http-listen 127.0.0.1:18889

REDIS_ADDR=127.0.0.1:6379 \
SESSIONSTREAM_BUS_TOPIC=sessionstream.chatdemo.redis \
SESSIONSTREAM_CONSUMER_ID=server-b \
go run ./examples/goja-redis-chatdemo-server/cmd/redis-host \
  serve chatbot serve --http-listen 127.0.0.1:18890
```

Open both URLs. Submitting a prompt on one process should publish events to Redis; both processes consume the events and fan out to their connected websocket clients.

## Redis-only CLI verbs

This example demonstrates multiple jsverb source sets:

```yaml
sources:
  - id: sites
    kind: jsverbs
    from:
      dir: ../goja-chatdemo-server/verbs
  - id: redis-tools
    kind: jsverbs
    from:
      dir: ./verbs

commands:
  - id: verbs
    type: builtin.jsverbs
    sources: [sites, redis-tools]
  - id: serve
    type: provider.command-set
    provider: go-go-goja-http
    name: serve
    sources: [sites]
```

The regular `sites` source provides the HTTP server verb. The Redis example adds a separate `redis` jsverb package for CLI tools that only make sense when the host is wired to Redis/Watermill:

```bash
REDIS_ADDR=127.0.0.1:6379 \
SESSIONSTREAM_BUS_TOPIC=sessionstream.chatdemo.redis \
go run ./examples/goja-redis-chatdemo-server/cmd/redis-host \
  verbs redis publish-trace \
  --session-id demo \
  --message-id cli-trace \
  --stage cli \
  --detail "injected from the Redis CLI"
```

Those CLI verbs create their own short-lived xgoja runtime. They affect a running browser/server only because the custom Go host injects the same Redis-backed event bus into both the CLI runtime and server runtime. In the plain in-memory chatdemo, the same verbs would publish only into the short-lived CLI process and would not reach an existing server.

## Why host services?

The generated runtime package exposes:

```go
NewBundle(xgojaruntime.Options{
  ConfigureServices: func(*app.HostServices) { ... },
})
```

The custom host uses that hook to set `sessionstream` provider host options:

```go
services.SetHostService(ssprovider.HostServiceKey, ssprovider.HostOptions{
  HubOptions: []sessionstream.HubOption{
    sessionstream.WithEventBus(pub, sub, sessionstream.WithBusTopic(topic)),
  },
})
```

The `sessionstream` provider reads the host service during module setup and passes the hub options into `require("sessionstream")`. JavaScript still calls normal APIs:

```js
const hub = ss.hub({ schemas })
hub.run()
await pub.publish("ChatTokensDelta", payload)
```

No Redis details leak into JavaScript.
